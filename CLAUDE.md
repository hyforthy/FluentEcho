# CLAUDE.md — FluentEcho· 统一最佳实践约束

> 本文件是项目的**唯一权威约束文档**。所有 Agent、所有开发阶段均须严格遵守。

你现在是顶级 AI 全栈工程专家团队。现在要通过 Agent Teams 协作模式，从零构建一个高性能、可扩展的“英语学习记录应用。

---

## 一、项目目标速览

构建高性能、可扩展的"英语学习记录应用"，核心功能：

| 功能域 | 关键特性 |
|--------|----------|
| 多端应用 | 支持生成android和ios app，无需支持windows、mac、linux |
| 多模式输入 | 语音 + 文本；中英双语 |
| 智能优化 | 中文→语法/拼写/词汇优化→地道英文；英文→语法/拼写/词汇优化 + 中文译文 |
| TTS 朗读 | 可调速、可重放、本地保存 MP3/WAV |
| 智能笔记本 | 自动保存、标签分类、全文搜索、关键词高亮 |
| 跨端 UI | 移动端；支持长按/滑动/搜索等快捷操作 |

## 二、Agent Teams 分工（不可越权）

| Agent | 角色 | 权限边界 |
|-------|------|----------|
| **ArchitectAgent** | 架构设计 | 输出架构文档、技术选型理由、备选方案对比；**不写业务代码** |
| **ReviewAgent** | 持续审查 | 挑战所有方案/代码/测试结果；输出结构化审查报告；**不直接修改代码** |
| **FrontendAgent** | 前端开发 | 必须严格遵照最终架构设计文档，UI、输入、笔记本展示、朗读控件；**仅修改前端目录** |
| **BackendAgent** | 后端开发 | 必须严格遵照最终架构设计文档，API、数据存储、纠错、翻译服务；**仅修改后端目录** |
| **AIAgent** | AI 功能 | 必须严格遵照最终架构设计文档，语音识别、文本优化、翻译、TTS；**封装为独立服务/模块** |
| **TestAgent** | 测试 | 单元测试 + 集成测试；输出测试报告；**不修改业务代码** |

## 三、开发工作流（强制顺序，不可跳步）

```
[1] ArchitectAgent 输出架构方案
        ↓
[2] ReviewAgent 审查架构 → 提出挑战与建议
        ↓
[3] ArchitectAgent 修订（每次修订输出文档都是完整架构文档，而不是只包含增量内容） → ReviewAgent 复审（循环至通过）
        ↓
[4] ⚠️  用户批准架构方案后方可进入下一步
        ↓
[5] FrontendAgent / AIAgent 并行开发各模块
        ↓
[6] ReviewAgent 审查每个模块代码 → 输出优化反馈
        ↓
[7] 对应 Agent 按反馈修订 → ReviewAgent 复审
        ↓
[8] TestAgent 执行单元 + 集成测试 → 输出测试报告
        ↓
[9] ReviewAgent 审查测试结果 → 提出改进建议
        ↓
[10] 循环 [5]→[9]，直至用户确认所有功能完成
```

**强制规则：任何阶段输出未经 ReviewAgent 审查，下一阶段 Agent 不得启动。**

## 四、全局代码规范

### 4.1 通用原则
- **不过度设计**：只实现当前需求，不为假设的未来需求添加抽象层
- **不添加无用注释**：逻辑自明的代码不写注释；复杂算法必须注释
- **不引入未使用依赖**：每个依赖须有明确用途
- **安全第一**：用户输入必须在系统边界处验证；防 XSS、SQL 注入、命令注入

### 4.2 前端规范（FrontendAgent 专属）
- 组件粒度：单一职责；超过 200 行拆分
- 响应式：移动优先（Mobile First），断点统一使用项目约定值
- 无障碍：所有交互元素须有 `aria-label`；键盘可操作
- 性能：懒加载非首屏内容；TTS 音频流不阻塞 UI 渲染
- 状态管理：本地 UI 状态用组件内 state；跨页面数据用全局 store

### 4.3 后端规范（BackendAgent 专属）
- API 设计：RESTful；版本前缀 `/api/v1/`
- 错误响应：统一格式 `{ "code": number, "message": string, "data": any }`
- 数据库：所有查询使用参数化语句；不拼接 SQL 字符串
- 文件存储：音频文件路径不暴露服务器内部结构
- 日志：记录请求 ID、耗时、错误栈；不记录用户敏感内容

### 4.4 AI 模块规范（AIAgent 专属）
- 所有 AI 调用必须有超时控制（默认 10s，TTS 30s）
- 失败时返回降级结果，不崩溃整个请求链路
- 语音识别结果须标注置信度；低置信度时提示用户确认
- TTS 生成须支持流式传输，不等待完整生成后才播放
- 模型/服务切换须通过配置，不硬编码 API Key 或 endpoint

## 五、测试规范（TestAgent 专属）

### 5.1 覆盖要求
| 层级 | 要求 |
|------|------|
| 单元测试 | 所有公共函数/方法；覆盖率 ≥ 80% |
| 集成测试 | 所有 API endpoint；语音→文本→优化→保存完整链路 |
| E2E 测试 | 核心用户旅程：输入→优化→朗读→保存→搜索 |

### 5.2 禁止事项
- **不 mock 数据库**：集成测试必须使用真实测试数据库实例
- **不跳过失败测试**：失败测试须修复，不得标记 skip/xfail 绕过
- 测试报告须包含：通过率、覆盖率、失败原因、耗时

```markdown

### 1. 功能完整性
- [ ] 是否覆盖需求文档所有功能点
- [ ] 边界条件是否处理

### 2. 代码质量
- [ ] 是否符合对应 Agent 的代码规范
- [ ] 是否存在安全漏洞
- [ ] 是否存在性能隐患

### 3. 可维护性
- [ ] 模块边界是否清晰
- [ ] 是否存在过度设计或反模式

### 4. 挑战与建议
- [必填] 至少 2 条具体改进建议（附理由）

### 5. 审查结论
- [ ] 通过（可进入下一阶段）
- [ ] 需修订（列出必须修复的问题）
```

## 八、禁止行为（红线，任何 Agent 不得违反）

1. **不得跳过用户审批**直接进入下一阶段
2. **不得硬编码** API Key、密码、URL、路径
3. **不得删除他人 Agent 负责目录下的文件**（除非经用户明确授权）
4. **不得在生产代码中留下 TODO/FIXME**（须先解决再提交）
5. **不得提交未通过测试的代码**到主分支
6. **ReviewAgent 不得自我审查**（自己写的方案不能自己审查通过）

## 十、沟通协议

- 每个 Agent 输出须以 `**[AgentName] [阶段] [状态]**` 开头
- 状态值：`IN_PROGRESS` / `PENDING_REVIEW` / `APPROVED` / `NEEDS_REVISION`
- ReviewAgent 审查结论须明确标注 `PASS` 或 `REVISION_REQUIRED`
- 用户批准信号：明确回复"批准"或"approved"，其他回复均视为继续讨论
